# QA項目を生成し、自動実行してPRに報告する

変更内容から QA項目を生成 → 自動で実行できる範囲を検証 → 満足したら PR コメントに「実施した項目」と「不安な点」を投稿する、までを自律的に行う。

**リポジトリ固有の QA 手順・テストコマンドがあればそれを優先する**（プロジェクト版 `/qa` が存在する場合はそちらが優先される）。

## フロー全体

1. 対象を決定（PR or ブランチ diff）
2. QA項目を生成し `~/.claude/tmp/qa-<id>.md` に保存
3. 自動実行できる項目を実行し、結果をファイルに書き戻す（`- [x]` / `- [ ]` / `- [~]`）
4. 満足できたら PR コメントを投稿。できなければチャットで報告

## 1. 対象の決定

- 引数にPR番号があれば（例: `/qa 123`）→ `gh pr diff <num>`、コメント先もそのPR
- なければ現ブランチ vs デフォルトブランチ（`git diff <default>...HEAD`）→ `gh pr view` で対応PRを特定
- フロント/別リポジトリにまたがる変更なら、関連リポジトリも確認

## 2. QA項目の生成（ファイル保存）

- `mkdir -p ~/.claude/tmp`、ファイル名は `qa-pr<番号>.md` / `qa-<TICKET-ID>.md` / `qa-<branch-slug>.md`（同名は上書き）
- 画面/機能/モジュールごとに見出し、各項目は tickable: `- [ ] **[観点]** 手順 → 期待: 結果`
- 観点を網羅（該当のみ）: 正常系 / 異常系 / 境界値 / 回帰 / データ保全(消失・truncate) / パフォーマンス
- 内部挙動のみで不変なら「挙動不変・回帰確認のみ」と明記し過剰生成しない

## 3. 自動実行（安いものから。ローカルで終わらせず E2E まで踏み込む）

実環境を使う段5の可否は後述の「実環境 E2E ポリシー」に従う（preview はガッツリ / staging は条件付き / 本番は自動操作なし）。

| 段 | 手段 | 用途 |
|---|---|---|
| 1 | ビルド / 型チェック / lint（リポジトリのコマンド） | 健全性 |
| 2 | テストスイート（リポジトリのテストランナーで関連テスト） | ロジック・回帰の大半はここで検証 |
| 3 | ローカル起動 + API/CLI 実行 | テストで届かない経路。認証はローカル手段で |
| 4 | DB / ストレージの実測確認 | 保存結果・サイズ・truncate 等 |
| 5 | **preview/staging で実 E2E**（ブラウザ操作=Playwright / 実 API / 実ストレージ） | UI・実環境依存・大容量など単体で届かない経路を実機で通す |

実行結果をファイルに反映: 成功 `- [x]`、失敗 `- [ ]`（失敗内容を追記）。

### 実環境 E2E ポリシー（段5。ローカルで届かないものは実環境で通す）

| 環境 | 方針 |
|---|---|
| **preview**（PRごとの使い捨て環境） | **ガッツリ自動・確認不要**。deploy / 大容量アップロード / データ投入 / UI操作 / 環境クラッシュ PoC まで踏み込んでよい（壊れても捨てる前提） |
| **staging**（共有・永続環境） | **軽い・非破壊は確認なし自動**（GET / route スモーク / 既存データでの参照系 E2E）。**重い・破壊的は事前確認**（deploy / 大容量アップロード・DL / DB・ストレージへのデータ投入 / 既存データを変える操作）→ AskUserQuestion で一言確認してから実行 |
| **本番**（production） | **自動操作しない**（read も含め原則触らない）。必要なら手動扱いで `- [~]` |

- UI操作（ブラウザ必須）は段5で **Playwright** を使い preview/staging で実行する。認証は project の E2E 手段（例: `reference_e2e_testing.md` の Playwright + DB 直確認）に従う。
- preview 環境の URL/起動方法が分かるなら最優先で段5を回す。無ければ staging で上表に沿って実行。
- 環境クラッシュ PoC（OOM 誘発等）は preview のみ可。staging/本番では回さない。

### それでも自動実行しない（必ず `- [~] 手動/未検証` + 理由）

- staging の「事前確認」対象で、**ユーザーが確認に応じない / 不在**のもの
- **本番環境への操作**（破壊・read 問わず原則触らない）
- 自動化コストが検証価値を上回る一回限りの重作業（理由を添えて手動に回す）

→ 自動実行できた分だけ `- [x]`。届かない分は結果を捏造せず `- [~]` とし「不安な点」に回す。

## 4. 満足の判定 → PRコメント投稿

**満足の条件**: 自動実行できた項目に致命的失敗が無く、未検証分を把握・明示できている。

満足したら対象PRにコメントを投稿（`gh pr comment <num> --body-file`）。簡潔・煽らない。

```markdown
## QA自動実行レポート

### ✅ 実施・確認した項目
- <項目> — <どう確認したか（test名 / API / DB確認）と結果>

### ⚠️ 不安な点・未検証
- <未検証の項目と理由（UI必須 / 環境制約 / 大容量）>
- <自動実行中に見つけた懸念・要手動確認>

チェックリスト全文: `~/.claude/tmp/qa-<id>.md`
```

**満足できない場合は投稿しない**: 環境が起動しない・致命的失敗・PRが無い等は、チャットでブロッカーを報告する。

## 原則

- **結果を捏造しない**。実際に実行した事実だけ `- [x]`。未実行は `- [~]` で理由を添える
- 本番への操作は自動でやらない。preview はガッツリ可、staging の重い/破壊的操作は事前確認（「実環境 E2E ポリシー」参照）
- リポジトリ固有の QA 規約 / テスト手順があれば優先する
- diff に無い機能の QA は作らない
- PRコメントは1論点ずつ簡潔に。チャットには保存先＋サマリだけ返し全文は貼らない
